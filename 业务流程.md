## 1. 认证与会话管理

### 1.1 注册 (Sign Up)

| 步骤 | 描述 |
|------|------|
| 1 | 前端 POST `/api/auth/signup` 携带 `{email, fullName, password}` |
| 2 | 校验：email 非空、fullName 非空、password ≥ 8 字符 |
| 3 | 首个注册用户自动获得 `admin` 角色；之后注册关闭（仅管理员可创建用户） |
| 4 | 密码使用 **bcrypt (cost=12)** 哈希，salt 字段置空 |
| 5 | 写入 `users` 表，创建 `user_sessions` 记录 |
| 6 | 返回 session token（随机 32 字节 → SHA256 存储），cookie 有效期 7 天 |

**业务规则**：
- `CanSelfRegister()` → 当 `users` 表行数 = 0 时允许注册
- Email 唯一约束

**涉及函数**：`RegisterUser()`, `hashPasswordBcrypt()`, `createSession()`, `createUserRecord()`

### 1.2 登录 (Login)

| 步骤 | 描述 |
|------|------|
| 1 | 前端 POST `/api/auth/login` 携带 `{email, password}` |
| 2 | 根据 email 查询用户（含 password_hash、password_salt） |
| 3 | `verifyPassword()`: 自动检测 bcrypt hash（`$2a$` 前缀）或 legacy SHA256 |
| 4 | 如果是 legacy SHA256 且验证成功 → 后台 goroutine `upgradePasswordHash()` 异步升级为 bcrypt |
| 5 | 校验 `is_active = TRUE` |
| 6 | 创建新 session → 返回 token cookie |

**业务规则**：
- 账户停用后无法登录
- 密码透明迁移：旧 SHA256 用户登录后自动升级为 bcrypt

**涉及函数**：`Login()`, `verifyPassword()`, `isBcryptHash()`, `upgradePasswordHash()`

### 1.3 会话验证 (Auth Middleware)

| 步骤 | 描述 |
|------|------|
| 1 | `requireAuth()` 中间件从 cookie 读取 session token |
| 2 | `GetUserBySessionToken()` → 根据 token_hash 查 `user_sessions` JOIN `users` |
| 3 | 校验：token 未过期 + 用户 is_active |
| 4 | 将 user 信息写入 Gin context |

### 1.4 登出 (Logout)

| 步骤 | 描述 |
|------|------|
| 1 | POST `/api/auth/logout` |
| 2 | 根据 token_hash 删除 `user_sessions` 记录 |
| 3 | 清除 cookie |

### 1.5 角色权限

| 角色 | 说明 | 权限范围 |
|------|------|----------|
| `admin` | 管理员 | 全部操作：用户管理、客户/SKU/仓库 CRUD、审计日志查看、UI 偏好设置 |
| `operator` | 操作员 | 文档操作：入库/出库/移库/调整/盘点 CRUD；库存项 CRUD |
| `viewer` | 只读 | 仅查看：列表查询、Dashboard |

**路由保护**：
- `protected` 组 → 需登录（任意角色）
- `operator` 组 → `requireRoles(admin, operator)`
- `admin` 组 → `requireRoles(admin)`

## 2. 用户管理

### 2.1 用户列表

- API: GET `/api/users` (admin only)
- 函数: `ListUsers()`
- 返回: 所有用户（id, email, fullName, role, isActive, createdAt）

- API: POST `/api/users` (admin only)
- 输入: `{email, fullName, password, role, isActive}`
- 函数: `CreateManagedUser()`
- 密码使用 bcrypt 哈希
- 管理员可直接指定角色和启用状态

### 2.3 修改用户权限

- API: PUT `/api/users/:id/access` (admin only)
- 输入: `{role, isActive}`
- 函数: `UpdateUserAccess()`
- 可切换角色、启用/停用账户

## 4. 入库 (Inbound)

### 4.1 文档状态机

```
DRAFT ──> CONFIRMED ──> [ARCHIVED]
  │                         ↑
  └──> CANCELLED            │
         │                  │
         └──────────────────┘ (可归档)
```

**文档状态 (status)**：
| 状态 | 说明 |
|------|------|
| `DRAFT` | 草稿，可编辑 |
| `CONFIRMED` | 已确认 → 触发库存写入 |
| `CANCELLED` | 已取消 → 触发库存回撤 |

**跟踪状态 (tracking_status)**：
```
SCHEDULED → ARRIVED → RECEIVING → RECEIVED
```
- 只能向前推进，不能回退
- 当 tracking_status 推到 `RECEIVED` 时，如果文档还是 DRAFT → 自动触发 Confirm

### 4.2 创建入库文档

- API: POST `/api/inbound-documents` (operator+)
- 函数: `CreateInboundDocument()`

| 步骤 | 描述 |
|------|------|
| 1 | 前端提交：客户、仓库、柜号、到货日期、存储区域、行项目（SKU + 预期数量 + 实收数量 + 卡板数） |
| 2 | 可选状态：`DRAFT` 或 `CONFIRMED`（创建时直接确认） |
| 3 | 写入 `inbound_documents` + `inbound_document_lines` |
| 4 | 如果 status = CONFIRMED → 立即执行确认流程（见 4.4） |

### 4.3 更新入库文档

- API: PUT `/api/inbound-documents/:id` (operator+)
- 函数: `UpdateInboundDocument()`
- **仅 DRAFT 状态**可更新
- 支持修改头信息 + 行项目（全量替换）
- **已确认文档**也允许有限修改（仅限头信息字段，如 documentNote, deliveryDate, containerNo），会对差异行做自动修正

### 4.4 确认入库文档 (核心流程)

- API: POST `/api/inbound-documents/:id/confirm` (operator+)
- 函数: `ConfirmInboundDocument()` → `confirmInboundDocumentTx()`

| 步骤 | 描述 |
|------|------|
| 1 | 校验：不能是 CANCELLED / 已 CONFIRMED |
| 2 | 加载所有行项目 |
| 3 | `ensureContainerVisitForInboundDocumentTx()` → 确保容器访问记录存在 |
| 4 | **对每一行：** |
| 4a | 计算 receivedQty（如果 = 0 则使用 expectedQty） |
| 4b | 同步当前余额桶：`inventory_items.quantity += receivedQty` |
| 4c | 写入 `stock_movements`（movement_type = "IN"）作为兼容总账 |
| 4d | 写入 `stock_ledger`（event_type = "RECEIVE"）作为 pallet v2 总账 |
| 4e | 创建 `receipt_lots` 记录，保留来源链 |
| 4f | 创建 `pallets`、`pallet_items`、`pallet_location_events` |
| 4g | 更新 `inbound_document_lines.movement_id` / `item_id` |
| 5 | 更新文档 status = CONFIRMED, tracking_status = RECEIVED, confirmed_at = now |

**当前 v2 口径**：
- `container_visits` 记录柜次来源
- `pallets + pallet_items` 记录真正的 pallet 数量与内容
- `inventory_items` 仅保留为当前余额桶 / 查询层

### 4.5 取消入库文档

- API: POST `/api/inbound-documents/:id/cancel` (operator+)
- 函数: `CancelInboundDocument()`

| 步骤 | 描述 |
|------|------|
| 1 | 校验：不能已取消 |
| 2 | **如果已确认**（有库存影响）： |
| 2a | 加载所有行项目及其 receipt_lots / pallets |
| 2b | 检查 receipt_lots / pallet_items 是否仍完整（没有被下游出库消耗） |
| 2c | 如果有部分消耗 → 拒绝取消 |
| 2d | 回撤：`inventory_items.quantity -= remainingQty` |
| 2e | 写入 REVERSAL 类型 `stock_movements` 与 `stock_ledger` |
| 2f | 回收 / 关闭对应 `receipt_lots`、`pallets`、`pallet_items` |
| 3 | **如果是 DRAFT** → 直接标记 CANCELLED |
| 4 | 写入 cancel_note, cancelled_at |

**业务规则**：已确认且有下游消耗的入库单不能取消

### 4.6 更新跟踪状态

- API: POST `/api/inbound-documents/:id/tracking-status` (operator+)
- 函数: `UpdateInboundDocumentTrackingStatus()`
- 只能向前推进：SCHEDULED → ARRIVED → RECEIVING → RECEIVED
- 推到 RECEIVED 时若文档为 DRAFT → 自动触发确认
- 已确认文档不能回退 tracking status

### 4.7 归档 / 复制

| 操作 | API | 函数 | 说明 |
|------|-----|------|------|
| 归档 | POST `.../archive` | `ArchiveInboundDocument()` | 设置 archived_at，从活动列表中隐藏 |
| 复制 | POST `.../copy` | `CopyInboundDocument()` | 从已有文档创建新 DRAFT，复制头信息和行项目 |

## 5. 出库 (Outbound)

### 5.1 文档状态机

```
DRAFT ──> CONFIRMED ──> [ARCHIVED]
  │                         ↑
  └──> CANCELLED            │
         │                  │
         └──────────────────┘
```

**跟踪状态 (tracking_status)**：
```
SCHEDULED → PICKING → PACKED → SHIPPED
```
- 推到 SHIPPED 时若 DRAFT → 自动触发 Confirm

### 5.2 创建出库文档

- API: POST `/api/outbound-documents` (operator+)
- 函数: `CreateOutboundDocument()`

| 步骤 | 描述 |
|------|------|
| 1 | 前端提交：packingListNo, orderRef, outDate, shipTo 信息, 行项目（itemId + 数量 + 卡板 + 重量） |
| 2 | 每行指定一个 `itemId` 作为当前余额入口；确认时会进一步分配到 pallet |
| 3 | 可选带 `pickAllocations`（指定从哪个 section/container 出货） |
| 4 | 写入 `outbound_documents` + `outbound_document_lines` + `outbound_pick_allocations` |
| 5 | 如果 status = CONFIRMED → 立即执行确认流程 |

### 5.3 确认出库文档 (核心流程)

- API: POST `/api/outbound-documents/:id/confirm` (operator+)
- 函数: `ConfirmOutboundDocument()` → `confirmOutboundDocumentTx()`

| 步骤 | 描述 |
|------|------|
| 1 | 校验：不能是 CANCELLED / 已 CONFIRMED |
| 2 | 加载所有行项目 |
| 3 | **对每一行：** |
| 3a | 加锁加载当前余额桶与可用 pallet 内容 |
| 3b | 解析 pick_allocations → `resolveOutboundLineAllocationsTx()` |
| 3c | 如果有手动分配 → 使用指定的 section/container 分配 |
| 3d | 如果无手动分配 → FIFO 自动分配 pallet / receipt lot |
| 3e | 校验库存充足：`availableQty >= allocatedQty` |
| 4 | **对每个 allocation：** |
| 4a | `inventory_items.quantity -= allocatedQty`（当前余额同步） |
| 4b | 写入 `stock_movements`（movement_type = "OUT"） |
| 4c | 消耗 `receipt_lots`（FIFO 扣减 remainingQty） |
| 4d | 扣减 `pallet_items.quantity`，必要时关闭 / 拆分 `pallets` |
| 4e | 写入 `stock_ledger`（event_type = "SHIP"） |
| 4f | 写入最终的 `outbound_pick_allocations` 与 `outbound_picks` |
| 5 | 更新文档 status = CONFIRMED, tracking_status = SHIPPED, confirmed_at = now |

**FIFO 自动分配逻辑**：
1. 先按 pallet / pallet_item 可用量尝试分配
2. 仍保留 `receipt_lots` 作为来源链与可回撤依据
3. 按 received_at / created_at 升序执行先入先出

### 5.4 取消出库文档

- API: POST `/api/outbound-documents/:id/cancel` (operator+)
- 函数: `CancelOutboundDocument()`

| 步骤 | 描述 |
|------|------|
| 1 | 校验：不能已取消 |
| 2 | **如果已确认**： |
| 2a | 遍历所有 pick_allocations |
| 2b | 回撤：`inventory_items.quantity += allocatedQty` |
| 2c | 写入 REVERSAL 类型 `stock_movements` 与 `stock_ledger` |
| 2d | 恢复 receipt_lots、pallet_items 与 pallet 状态 |
| 3 | **如果是 DRAFT** → 直接标记 CANCELLED |

### 5.5 更新跟踪状态

- API: POST `/api/outbound-documents/:id/tracking-status` (operator+)
- 函数: `UpdateOutboundDocumentTrackingStatus()`
- SCHEDULED → PICKING → PACKED → SHIPPED
- SHIPPED 时若 DRAFT → 自动确认

### 5.6 归档 / 复制

与入库相同模式：
- `ArchiveOutboundDocument()` → 设置 archived_at
- `CopyOutboundDocument()` → 从已有文档创建新 DRAFT

## 6. 库内移库 (Transfer)

### 6.1 状态

单一状态 `POSTED`（一旦创建即生效）

### 6.2 创建移库

- API: POST `/api/transfers` (operator+)
- 函数: `CreateInventoryTransfer()`

| 步骤 | 描述 |
|------|------|
| 1 | 输入：transferNo(可自动生成), notes, lines |
| 2 | 每行：sourceItemId + quantity + toLocationId + toStorageSection |
| 3 | 开始事务 |
| 4 | **对每一行：** |
| 4a | 加锁查询源 item 与相关 pallet 内容 |
| 4b | 校验库存充足 |
| 4c | 源 item 扣减：`quantity -= transferQty` |
| 4d | `findOrCreateTransferDestinationItem()` → 目标 item 增加：`quantity += transferQty` |
| 4e | 写入 2 条 `stock_movements`：TRANSFER_OUT + TRANSFER_IN |
| 4f | 写入 `stock_ledger`（event_type = "TRANSFER"） |
| 4g | 迁移对应 pallet 到新位置，必要时拆分 child pallet |
| 4h | 迁移 receipt_lots 到新位置 |
| 5 | 写入 `inventory_transfers` + `inventory_transfer_lines` |
| 6 | 提交事务 |

**业务规则**：
- 同一个 SKU+Customer+Location+Section 组合可能已存在 item → 合并
- 如果不存在 → 新建 destination item
- Transfer 是即时生效的，没有草稿/确认流程

**当前 v2 口径**：
- `inventory_items` 继续同步为当前余额
- pallet 的真实位置变化写入 `pallet_location_events`
- pallet 主线与旧流水同时保留，便于查询与回撤

### 6.3 查询

- API: GET `/api/transfers` (任意登录用户)
- 函数: `ListInventoryTransfers()`
- 返回 transfer + lines（含 from/to location + section 信息）
- `routes` 字段拼接所有不同的路径（如 "仓库A/S01 -> 仓库B/S02"）

## 7. 库存调整 (Adjustment)

### 7.1 状态

单一状态 `POSTED`（即时生效）

### 7.2 创建调整

- API: POST `/api/adjustments` (operator+)
- 函数: `CreateInventoryAdjustment()`

| 步骤 | 描述 |
|------|------|
| 1 | 输入：adjustmentNo(可自动生成), reasonCode, notes, lines |
| 2 | 每行：itemId + adjustQty（正/负均可） + lineNote |
| 3 | 开始事务 |
| 4 | **对每一行：** |
| 4a | 加锁查询 item |
| 4b | 记录 beforeQty |
| 4c | 校验调整后数量 ≥ 0 |
| 4d | `inventory_items.quantity += adjustQty` |
| 4e | afterQty = beforeQty + adjustQty |
| 4f | 写入 `stock_movements`（movement_type = "ADJUST"） |
| 4g | 写入 `stock_ledger`（event_type = "ADJUST"） |
| 5 | 写入 `inventory_adjustments` + `inventory_adjustment_lines` |
| 6 | 提交事务 |

**当前 v2 口径**：
- 调整优先同步到 pallet 内容；没有可挂接 pallet 时再作为余额修正处理
- 调整历史同时保留在旧流水与 `stock_ledger`

### 7.3 查询

- API: GET `/api/adjustments` (任意登录用户)
- 函数: `ListInventoryAdjustments()`
- 返回 adjustment + lines（含 before/adjust/after 数量）

## 8. 盘点 (Cycle Count)

### 8.1 状态

单一状态 `POSTED`（即时生效，差异自动写入）

### 8.2 创建盘点

- API: POST `/api/cycle-counts` (operator+)
- 函数: `CreateCycleCount()`

| 步骤 | 描述 |
|------|------|
| 1 | 输入：countNo(可自动生成), notes, lines |
| 2 | 每行：itemId + countedQty + lineNote |
| 3 | 开始事务 |
| 4 | **对每一行：** |
| 4a | 加锁查询 item → 获取 systemQty |
| 4b | varianceQty = countedQty - systemQty |
| 4c | 如果有差异 → `inventory_items.quantity = countedQty` |
| 4d | 写入 `stock_movements`（movement_type = "COUNT"，quantity_change = varianceQty） |
| 4e | 写入 `stock_ledger`（event_type = "COUNT"） |
| 5 | 写入 `cycle_counts` + `cycle_count_lines` |
| 6 | 提交事务 |

**业务规则**：
- 即使差异为 0 也要记录（审计需要）
- 直接覆盖系统数量为实盘数量

**当前 v2 口径**：
- 盘点结果会尽可能同步到 pallet 内容
- 同时保留 `inventory_items` 的聚合余额更新

### 8.3 查询

- API: GET `/api/cycle-counts` (任意登录用户)
- 函数: `ListCycleCounts()`
- 返回 count + lines（含 system/counted/variance 数量）

## 9. 审计日志 (Audit Log)

### 9.1 记录时机

**所有写操作结束后**，在 API handler 层调用 `writeAuditLog()`。

记录的操作类型包括：

| action | entity_type | 触发场景 |
|--------|-------------|----------|
| CREATE | customer | 创建客户 |
| UPDATE | customer | 更新客户 |
| DELETE | customer | 删除客户 |
| CREATE | sku_master | 创建 SKU |
| UPDATE | sku_master | 更新 SKU |
| DELETE | sku_master | 删除 SKU |
| CREATE | location | 创建仓库 |
| UPDATE | location | 更新仓库 |
| DELETE | location | 删除仓库 |
| CREATE | inbound_document | 创建入库单 |
| UPDATE | inbound_document | 更新入库单 |
| CONFIRM | inbound_document | 确认入库单 |
| CANCEL | inbound_document | 取消入库单 |
| ARCHIVE | inbound_document | 归档入库单 |
| COPY | inbound_document | 复制入库单 |
| TRACKING_STATUS | inbound_document | 更新入库跟踪状态 |

## 14. 转运 / Cross-Dock（集装箱直接转运）

### 14.1 业务场景

集装箱到达仓库后**不入库存储**，货物直接转发到下一个目的地。典型情况：
- 客户 A 的货到港 → 经仓库拆柜后直接装车发往客户 A 的门店
- 整柜或部分卡板直接从入库码头转到出库码头

### 14.2 Pallet 状态扩展

原状态：`STORED / SHIPPED / DAMAGED / HOLD / DISPOSED`

新增：**`CROSS_DOCK`**（转运中 — 货物到达但不入库，等待直接出发）

完整生命周期：
```
              ┌─────────── CROSS_DOCK ─────────┐
              │       (不入库，直接出)           │
              ▼                                 ▼
入库 ──> [STORED] ──> SHIPPED                SHIPPED
              │
              ├──> HOLD
              ├──> DAMAGED
              └──> DISPOSED
```

### 14.3 Inbound 文档扩展

`inbound_documents` 新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `is_cross_dock` | BOOLEAN DEFAULT FALSE | 标记此入库单为转运单 |
| `cross_dock_ship_to_name` | VARCHAR(160) | 转运目的地名称 |
| `cross_dock_ship_to_address` | VARCHAR(255) | 转运目的地地址 |
| `cross_dock_carrier_name` | VARCHAR(160) | 转运承运商 |
| `cross_dock_outbound_id` | BIGINT | 自动生成的出库单 ID（反向引用） |

### 14.4 转运确认流程

- API: POST `/api/inbound-documents/:id/confirm` (operator+)
- 当 `is_cross_dock = TRUE` 时触发特殊流程

| 步骤 | 描述 |
|------|------|
| 1 | 校验同正常入库 |
| 2 | 创建 pallets，**status = `CROSS_DOCK`**（而非 STORED） |
| 3 | 创建 pallet_items（数量来自入库行） |
| 4 | 写入 `stock_ledger` (event_type = "RECEIVE") |
| 5 | **自动生成出库单**（outbound_document）携带转运目的地信息 |
| 6 | **自动确认出库单** → 生成 outbound_picks 引用同批 pallets |
| 7 | pallets status 更新为 `SHIPPED`，shipped_at = now |
| 8 | 写入 `stock_ledger` (event_type = "SHIP") |
| 9 | 更新 inbound 文档的 `cross_dock_outbound_id` 反向引用 |
| 10 | 入库文档 status = CONFIRMED, tracking_status = RECEIVED |

**一次确认 = 入库 + 出库双写**，操作员只需要操作一步。

### 14.5 转运取消

| 情况 | 处理 |
|------|------|
| 尚未确认（DRAFT） | 正常取消 inbound |
| 已确认 | 同时回撤 inbound + 自动生成的 outbound，恢复 pallets |

### 14.6 部分转运

如果只有部分卡板需要转运，其余入库：

| 步骤 | 描述 |
|------|------|
| 1 | 创建入库单时标记 `is_cross_dock = TRUE` |
| 2 | 行项目中标记哪些需要转运（新增 `is_cross_dock_line` 字段） |
| 3 | 确认时：转运行 → pallets status `CROSS_DOCK → SHIPPED`；普通行 → pallets status `STORED` |
| 4 | 自动生成的出库单只包含转运行 |

### 14.7 计费特殊处理

| 费项 | 转运处理 |
|------|----------|
| 入库费 (container fee) | **照常收取** — 柜到仓就收费 |
| 缠膜费 (wrapping fee) | **可选** — 如果需要重新打包则收取 |
| 仓储费 (storage fee) | **免收或按小时** — 不占仓位；pallets.status 从未进入 STORED |
| 出库费 (outbound fee) | **照常收取** — 货从仓库出发 |

通过 `pallets.status` 历史判断是否走了 CROSS_DOCK 路径来区分计费。

### 14.8 schema 变更（Pallet-Centric 模型追加）

```sql
-- pallets 表 status 增加 CROSS_DOCK 值
-- pallets.status: STORED / CROSS_DOCK / SHIPPED / DAMAGED / HOLD / DISPOSED

-- inbound_documents 新增字段
ALTER TABLE inbound_documents
  ADD COLUMN is_cross_dock         BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN cross_dock_ship_to_name    VARCHAR(160) DEFAULT NULL,
  ADD COLUMN cross_dock_ship_to_address VARCHAR(255) DEFAULT NULL,
  ADD COLUMN cross_dock_carrier_name    VARCHAR(160) DEFAULT NULL,
  ADD COLUMN cross_dock_outbound_id     BIGINT DEFAULT NULL;

-- inbound_document_lines 新增字段（部分转运时标记）
ALTER TABLE inbound_document_lines
  ADD COLUMN is_cross_dock_line BOOLEAN NOT NULL DEFAULT FALSE;
```

### 14.9 与方案 A（双单据手动操作）的对比

| 维度 | 方案 A（手动双单据） | 方案 B（Cross-Dock 一等公民） |
|------|---------------------|-------------------------------|
| 操作步骤 | 创建入库 → 确认 → 创建出库 → 确认（4步） | 创建入库(标记转运) → 确认（2步） |
| Pallet 状态 | STORED → SHIPPED（经历虚假的"在库"） | CROSS_DOCK → SHIPPED（真实反映） |
| 计费准确性 | 可能误算仓储费 | 通过状态区分，不算仓储费 |
| 审计链 | 入库单和出库单独立，关系不明显 | inbound.cross_dock_outbound_id 明确关联 |
| 报表 | 需手动排除"秒存秒出"的数据 | 直接按 is_cross_dock 过滤 |
| 代码复杂度 | 无需新代码 | 需要新增确认分支逻辑 |

## 15. 渐进式收货 / Progressive Receiving

### 15.1 业务场景

集装箱到仓时**只知道 SKU 和预期总数量**（来自 packing list），但：
- 不知道会打成多少个 pallet
- 不知道每个 pallet 上具体有什么、有多少
- 要**卸完货、清点、打板之后才能确定**

这是仓库最常见的收货方式。当前"一步确认"模式要求提前知道所有信息，无法适应这个场景。

### 15.2 与现有"一步确认"模式的对比

| 维度 | 一步确认（现有） | 渐进式收货（新增） |
|------|-----------------|-------------------|
| 适用场景 | 预打好板的标准货物，数量明确 | 散货柜、需要仓库现场打板 |
| Pallet 信息 | 确认时必须全部确定 | 卸货过程中逐板录入 |
| 确认时机 | 创建/确认一步到位 | 先确认文档 → 逐板收货 → 完成 |
| 收货进度 | 无（要么全确认，要么未确认） | 实时可见：已收 X / 预期 Y |

### 15.3 Tracking Status 新语义

已有的 tracking_status 天然适配这个流程，赋予新的业务含义：

```
SCHEDULED ──> ARRIVED ──> RECEIVING ──> RECEIVED
    │             │            │            │
    │             │            │            └─ 所有板已录入，文档完成
    │             │            └────────────── 正在卸柜，逐板录入中
    │             └─────────────────────────── 柜已到仓，等待卸货
    └───────────────────────────────────────── 柜已排期，尚未到达
```

### 15.4 文档确认方式扩展

`inbound_documents` 新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `receiving_mode` | VARCHAR(32) DEFAULT 'IMMEDIATE' | `IMMEDIATE` = 一步确认（现有）；`PROGRESSIVE` = 渐进式收货 |

### 15.5 渐进式收货完整流程

#### 阶段 1：创建入库单

- API: POST `/api/inbound-documents`
- 与现有流程相同，提交 SKU + 预期总数量
- 设置 `receiving_mode = 'PROGRESSIVE'`
- **行项目只填 expected_qty，不填 received_qty 和 pallets**
- 文档状态：`DRAFT`

#### 阶段 2：确认入库单（开放收货）

- API: POST `/api/inbound-documents/:id/confirm`
- **不创建 pallets，不写库存**（与 IMMEDIATE 模式的关键区别）
- 仅更新：status = `CONFIRMED`
- 文档进入"等待收货"状态

#### 阶段 3：柜到达 → 开始卸货

- 操作员更新 tracking_status → `ARRIVED` → `RECEIVING`
- 进入逐板录入阶段

#### 阶段 4：逐板录入（核心新 API）

- API: **POST `/api/inbound-documents/:id/receive-pallet`** (operator+)
- 函数: `ReceiveInboundPallet()`

单次调用处理**一个 pallet**：

| 步骤 | 描述 |
|------|------|
| 1 | 输入：`{ palletNo (可选自动生成), storageSection, items: [{ skuMasterId, quantity }] }` |
| 2 | 开始事务 |
| 3 | 校验：文档 status = CONFIRMED，tracking_status 为 ARRIVED 或 RECEIVING |
| 4 | 自动将 tracking_status 推进到 **RECEIVING**（如果还在 ARRIVED） |
| 5 | 创建 `pallets` 记录（status = STORED, source_inbound_document_id = 文档 ID） |
| 6 | 为 pallet 上的每个 SKU 创建 `pallet_items` 记录 |
| 7 | 更新对应 `inbound_document_lines.received_qty += quantity`（累加） |
| 8 | 写入 `stock_ledger`（event_type = "RECEIVE"） |
| 9 | 提交事务 |
| 10 | 返回创建的 pallet 信息 + 文档的收货进度 |

**输入示例**（混合板）：
```json
{
  "palletNo": "PLT-2026-0042",
  "storageSection": "A01",
  "items": [
    { "skuMasterId": 15, "quantity": 200 },
    { "skuMasterId": 23, "quantity": 80 }
  ]
}
```

#### 阶段 5：查看收货进度

- API: GET `/api/inbound-documents/:id` (现有)
- 返回数据新增：

| 字段 | 说明 |
|------|------|
| `receivingMode` | "PROGRESSIVE" |
| `receivedPalletCount` | 已录入的 pallet 数 |
| 每行的 `expectedQty` vs `receivedQty` | 实时对比，差异一目了然 |
| `receivedPallets` | 该文档关联的所有 pallets 列表 |

#### 阶段 6：完成收货

- API: **POST `/api/inbound-documents/:id/complete-receiving`** (operator+)
- 函数: `CompleteInboundReceiving()`

| 步骤 | 描述 |
|------|------|
| 1 | 校验：tracking_status = RECEIVING |
| 2 | 更新 tracking_status = **RECEIVED** |
| 3 | 记录 received_at = now |
| 4 | 计算最终差异：每行 `variance = receivedQty - expectedQty` |
| 5 | 如果有差异 → 记录到审计日志（variance report） |
| 6 | 返回完成后的文档（含差异报告） |

**允许超收和少收**：
- received_qty > expected_qty → 超收，记录差异
- received_qty < expected_qty → 少收，记录差异
- 均不阻止完成，仅记录

#### 可选：删除/修正已录入的 Pallet

- API: **DELETE `/api/inbound-documents/:id/pallets/:palletId`** (operator+)
- 仅在 tracking_status = RECEIVING 时允许
- 回撤该 pallet 的库存写入
- 更新 `inbound_document_lines.received_qty -= quantity`

### 15.6 两种模式的时序对比

#### IMMEDIATE 模式（现有）
```
创建入库单(DRAFT, 含全部信息)
  → 确认 → pallets 全部创建 + 库存全部写入
  → tracking_status 直接跳到 RECEIVED
  → 完成
```

#### PROGRESSIVE 模式（新增）
```
创建入库单(DRAFT, 只有 expected_qty)
  → 确认(不写库存)
  → tracking: SCHEDULED → ARRIVED → RECEIVING
  → receive-pallet (重复 N 次，每次一板)
      → 逐个创建 pallet + pallet_items + stock_ledger
      → received_qty 累加
  → complete-receiving
  → tracking: RECEIVED
  → 完成（含差异报告）
```

### 15.7 新增 API 端点汇总

| Method | Path | 角色 | 说明 |
|--------|------|------|------|
| POST | `/api/inbound-documents/:id/receive-pallet` | operator+ | 录入一个 pallet |
| POST | `/api/inbound-documents/:id/complete-receiving` | operator+ | 完成收货 |
| DELETE | `/api/inbound-documents/:id/pallets/:palletId` | operator+ | 删除已录入的 pallet |
| GET | `/api/inbound-documents/:id/pallets` | 登录 | 查看已录入的所有 pallets |

### 15.8 schema 变更

```sql
-- inbound_documents 新增字段
ALTER TABLE inbound_documents
  ADD COLUMN receiving_mode VARCHAR(32) NOT NULL DEFAULT 'IMMEDIATE';
  -- IMMEDIATE = 一步确认（现有行为）
  -- PROGRESSIVE = 渐进式收货
```

`pallets.source_inbound_document_id` 已有 → 可直接关联查询某入库单的所有已收 pallets。

`inbound_document_lines.received_qty` 已有 → 在 PROGRESSIVE 模式下从 0 逐步累加。

**无需新增表，利用现有 pallet-centric 结构即可完整支撑。**

## 16. 整柜暂存 / Container Relay

### 16.1 业务场景

集装箱到达仓库后**不拆封、不卸货**，整柜停放在仓库或堆场，停留一段时间后**整柜直接运走**。

典型情况：
- 港口提柜后暂存在仓库堆场，等客户安排提货
- 货物只是中转，等下一程运输安排
- 客户暂时没有卸货需求，先存柜再说

**关键特征**：全程不产生 Pallet，操作对象是 Container 本身。

### 16.2 与其他入库模式的对比

| 维度 | 一步确认 | 渐进式收货 | Cross-Dock | 整柜暂存 |
|------|---------|-----------|------------|----------|
| 拆柜 | ✅ | ✅ | ✅ | ❌ |
| 产生 Pallet | ✅ | ✅ 逐板 | ✅ 即走 | ❌ |
| 存储单位 | Pallet | Pallet | Pallet(短暂) | Container |
| 计费基准 | 板·周 | 板·周 | 板(进出费) | 柜·天 |

### 16.3 数据模型

整柜暂存不经过 Pallet 层，需要一个独立的 **Container 停留记录**：

```sql
CREATE TABLE IF NOT EXISTS container_stays (
  id                   BIGINT NOT NULL AUTO_INCREMENT,
  inbound_document_id  BIGINT NOT NULL,
  customer_id          BIGINT NOT NULL,
  location_id          BIGINT NOT NULL,
  storage_section      VARCHAR(16) NOT NULL DEFAULT 'YARD',
  container_no         VARCHAR(120) NOT NULL,
  container_type       VARCHAR(32) NOT NULL DEFAULT '40HQ',
                       -- 20GP / 40GP / 40HQ / 45HQ / OTHER
  seal_no              VARCHAR(64) DEFAULT NULL,
  status               VARCHAR(32) NOT NULL DEFAULT 'PARKED',
                       -- PARKED / RELEASED / CANCELLED
  arrived_at           TIMESTAMP NULL DEFAULT NULL,
  released_at          TIMESTAMP NULL DEFAULT NULL,
  release_carrier      VARCHAR(160) DEFAULT NULL,
  release_destination  VARCHAR(255) DEFAULT NULL,
  release_note         TEXT DEFAULT NULL,
  created_at           TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at           TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_container_stays_inbound (inbound_document_id),
  KEY idx_container_stays_customer (customer_id),
  KEY idx_container_stays_status (status),
  KEY idx_container_stays_container_no (container_no),
  CONSTRAINT fk_container_stays_inbound
    FOREIGN KEY (inbound_document_id) REFERENCES inbound_documents (id),
  CONSTRAINT fk_container_stays_customer
    FOREIGN KEY (customer_id) REFERENCES customers (id),
  CONSTRAINT fk_container_stays_location
    FOREIGN KEY (location_id) REFERENCES storage_locations (id)
);
```

### 16.4 Inbound 文档扩展

`receiving_mode` 新增值：

| 值 | 说明 |
|------|------|
| `IMMEDIATE` | 一步确认（现有） |
| `PROGRESSIVE` | 渐进式收货 |
| `CONTAINER_RELAY` | 整柜暂存，不拆封 |

### 16.5 完整流程

#### 阶段 1：创建入库单

- API: POST `/api/inbound-documents`
- `receiving_mode = 'CONTAINER_RELAY'`
- 必填：customer_id, location_id, container_no
- 行项目**可选**（packing list 信息，仅作参考记录，不影响库存）
- 可补充：container_type, seal_no

#### 阶段 2：确认入库单（柜到达）

- API: POST `/api/inbound-documents/:id/confirm`
- 当 `receiving_mode = 'CONTAINER_RELAY'` 时触发特殊流程：

| 步骤 | 描述 |
|------|------|
| 1 | **不创建 pallets，不写 pallet_items** |
| 2 | 创建 `container_stays` 记录（status = PARKED, arrived_at = now） |
| 3 | 写入 `stock_ledger`（event_type = "CONTAINER_IN"，pallet_id = NULL，quantity_change = 0） |
| 4 | 更新文档 status = CONFIRMED, tracking_status = RECEIVED |

#### 阶段 3：整柜释放（运走）

- API: **POST `/api/container-stays/:id/release`** (operator+)
- 函数: `ReleaseContainerStay()`

| 步骤 | 描述 |
|------|------|
| 1 | 输入：`{ carrierName, destination, note }` |
| 2 | 校验 status = PARKED |
| 3 | 更新 `container_stays`: status = RELEASED, released_at = now |
| 4 | 写入 `stock_ledger`（event_type = "CONTAINER_OUT"） |
| 5 | 写入审计日志 |

**不产生出库单**（因为没有 pallet/SKU 级别的出库操作），或者可选自动生成一张标记性出库单。

#### 可选：中途转为拆柜

如果客户改主意要拆柜：

| 步骤 | 描述 |
|------|------|
| 1 | 取消 container_stay（status = CANCELLED） |
| 2 | 修改入库单 `receiving_mode` → `PROGRESSIVE` |
| 3 | 开始逐板收货流程（第 15 节） |

### 16.6 Container Stay 状态机

```
PARKED ──────> RELEASED      (正常运走)
   │
   └────────> CANCELLED      (改为拆柜 / 入库单取消)
```

### 16.7 计费

整柜暂存的计费维度与 pallet 完全不同：

| 费项 | 计算方式 |
|------|----------|
| 柜场停放费 | **按柜·天**：(released_at - arrived_at) × 日费率 |
| 柜型加价 | 40HQ 比 20GP 贵 |
| 进场费 | 每柜一次性 |
| 出场费 | 每柜一次性 |

`customer_rate_cards` 表扩展：

```sql
ALTER TABLE customer_rate_cards
  ADD COLUMN container_yard_fee_per_day_20  DECIMAL(12,2) NOT NULL DEFAULT 15.00,
  ADD COLUMN container_yard_fee_per_day_40  DECIMAL(12,2) NOT NULL DEFAULT 25.00,
  ADD COLUMN container_entry_fee            DECIMAL(12,2) NOT NULL DEFAULT 100.00,
  ADD COLUMN container_exit_fee             DECIMAL(12,2) NOT NULL DEFAULT 100.00;
```

### 16.9 仓库全景：四种入库模式一览

```
集装箱到达
    │
    ├─ 不拆封 ──────────────── Container Relay（整柜暂存）
    │                              → container_stays
    │                              → 按柜·天计费
    │
    └─ 拆封卸货
         │
         ├─ 信息完整 ──────── Immediate（一步确认）
         │                      → pallets 一次性全部创建
         │
         ├─ 逐板清点 ──────── Progressive（渐进式收货）
         │                      → pallets 逐个创建
         │
         └─ 不入库直接走 ──── Cross-Dock（转运）
                                → pallets 状态 CROSS_DOCK → SHIPPED
```

## 附录：数据流总览

### 旧模型 (Item-Centric，逐步下线)

```
入库确认 → inventory_items.quantity += N
         → stock_movements (IN)
         → receipt_lots (FIFO 批次)

出库确认 → receipt_lots FIFO 扣减
         → inventory_items.quantity -= N
         → stock_movements (OUT)
         → outbound_pick_allocations

移库     → source item.quantity -= N
         → dest item.quantity += N
         → stock_movements (TRANSFER_OUT + TRANSFER_IN)
         → receipt_lots 迁移

调整     → inventory_items.quantity += adjustQty
         → stock_movements (ADJUSTMENT)

盘点     → inventory_items.quantity = countedQty
         → stock_movements (COUNT, variance)
```

### 当前执行主线 (Pallet-Centric v2)

```
入库确认 → container_visits
         → receipt_lots
         → 创建 pallets (status=OPEN/STORED) + pallet_items
         → stock_movements (IN)
         → stock_ledger (RECEIVE)

出库确认 → outbound_pick_allocations / outbound_picks
         → pallet_items.quantity -= N
         → receipt_lots FIFO 扣减
         → inventory_items.quantity -= N
         → stock_movements (OUT)
         → stock_ledger (SHIP)

移库     → pallet_location_events
         → 必要时拆分 child pallet
         → inventory_items 源减目标增
         → stock_movements (TRANSFER_OUT + TRANSFER_IN)
         → stock_ledger (TRANSFER)

调整     → inventory_items.quantity += adjustQty
         → pallet_items 视情况同步
         → stock_movements (ADJUST)
         → stock_ledger (ADJUST)

盘点     → inventory_items.quantity = countedQty
         → pallet_items 视情况同步
         → stock_movements (COUNT)
         → stock_ledger (COUNT)

库存查询 → 以 inventory_items 作为聚合查询层
         → 以 pallets / pallet_items / pallet_location_events 作为 pallet 真相层
```
